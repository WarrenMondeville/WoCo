# ET 框架的 Fiber 与 Actor 模型详解

> 本文档详细分析 ET 框架中的 Fiber（纤程）与 Actor 模型，包括其架构设计、消息机制、调度策略及使用方式。

---

## 一、概述

ET 框架采用 **Fiber（纤程）+ Actor** 的并发模型，实现逻辑单线程化与安全的跨执行单元通信：

- **Fiber**：最小的并发调度单元，类似 Erlang 进程、Skynet 的 lua 虚拟机，每个 Fiber 拥有独立的 Scene、Entity 树和消息队列。
- **Actor**：可接收消息的 Entity，通过挂载 `MailBoxComponent` 成为 Actor，消息按队列顺序处理。
- **核心原则**：同一时刻，每个 Fiber 内只有一个逻辑在执行；不同 Fiber 之间**仅能通过消息通信**，禁止直接引用或调用。

## 二、Fiber（纤程）

### 2.1 Fiber 的定义与职责

```csharp
// Unity/Assets/Scripts/Core/Fiber/Fiber.cs
public class Fiber: IDisposable
{
    public int Id;                    // 纤程唯一 ID
    public int Zone;                  // 分区 ID（多区服）
    public Scene Root { get; }        // 根 Scene，所有 Entity 的父节点
    public EntitySystem EntitySystem { get; }
    public Mailboxes Mailboxes { get; }   // 本 Fiber 内所有 Actor 的信箱集合
    public ThreadSynchronizationContext ThreadSynchronizationContext { get; }
    // ...
}
```

每个 Fiber 具有：

| 组件 | 作用 |
|------|------|
| `Root` | 根 Scene，承载该 Fiber 的 Entity 树 |
| `EntitySystem` | Entity 生命周期与 System 驱动 |
| `Mailboxes` | 管理本 Fiber 内所有 `MailBoxComponent`，用于消息路由 |
| `ThreadSynchronizationContext` | 协程/异步回调的调度上下文，保证在「Fiber 线程」执行 |

### 2.2 Fiber 的创建与初始化

```csharp
// FiberManager.Create 流程
public async ETTask<int> Create(SchedulerType schedulerType, int fiberId, int zone, SceneType sceneType, string name)
{
    Fiber fiber = new(fiberId, zone, sceneType, name);
    this.fibers.TryAdd(fiberId, fiber);
    this.schedulers[(int)schedulerType].Add(fiberId);  // 加入调度器

    // 在 Fiber 的线程同步上下文中执行初始化
    fiber.ThreadSynchronizationContext.Post(async () =>
    {
        await EventSystem.Instance.Invoke<FiberInit, ETTask>((long)sceneType, new FiberInit() { Fiber = fiber });
    });
    return fiberId;
}
```

要点：

1. 根据 `SceneType` 分发到对应的 `FiberInit_XXX` Handler。
2. 初始化必须在 **该 Fiber 的上下文** 中执行，以保证组件挂在正确的 Scene 上。
3. `FiberManager` 不允许外部直接持有 Fiber 引用，仅通过消息访问（见注释：「不允许直接获取其它 Fiber 引用」）。

### 2.3 SceneType 与典型 Fiber

| SceneType | 说明 | 典型用途 |
|-----------|------|----------|
| Main | 主纤程，进程唯一 | 入口、全局初始化 |
| NetInner | 进程内网络通信 | 跨进程消息转发 |
| Realm | 登录服 | 账号验证、Gate 分配 |
| Gate | 网关 | 连接管理、消息转发 |
| Map | 地图服 | 场景逻辑、Unit 管理 |
| Location | 位置服 | Actor Location 寻址 |
| Match | 匹配服 | 对战匹配 |
| Room / RoomRoot | 房间 | 对局逻辑 |
| **NetClient** | 客户端网络纤程 | 网络连接、Realm/Gate 通信 |
| **Robot** | 机器人 | 压测、AI |

### 2.4 调度器（Scheduler）

ET 提供三种调度类型：

```csharp
public enum SchedulerType
{
    Main,       // 主线程
    Thread,     // 独立线程
    ThreadPool, // 线程池
}
```

#### MainThreadScheduler

- 所有 Main Fiber 在主线程运行。
- 每帧依次调用各 Fiber 的 `Update()` 和 `LateUpdate()`。
- 与 Unity 主循环配合，用于 UI、主场景逻辑。

#### ThreadPoolScheduler

- 使用 `Environment.ProcessorCount` 个线程轮询 Fiber。
- 每个线程从队列取出 Fiber ID，执行 `Update` + `LateUpdate`，然后放回队列。
- 适合 CPU 密集型、无 UI 依赖的 Fiber（如 Gate、Map、NetClient）。

#### 编译条件影响

```csharp
#if ENABLE_VIEW && UNITY_EDITOR
    // Editor：Thread/ThreadPool 都退化为 MainThreadScheduler，便于调试
    this.schedulers[(int)SchedulerType.Thread] = this.mainThreadScheduler;
    this.schedulers[(int)SchedulerType.ThreadPool] = this.mainThreadScheduler;
#else
    this.schedulers[(int)SchedulerType.Thread] = new ThreadScheduler(this);
    this.schedulers[(int)SchedulerType.ThreadPool] = new ThreadPoolScheduler(this);
#endif
```

## 三、进程内消息通信（Actor 模型）

### 3.1 ActorId 与 Address

```
ActorId = Address + InstanceId
Address = (Process, Fiber)   // 定位到 Fiber
InstanceId                 // 定位到 Fiber 内某 Entity（通过 MailBoxComponent）
```

```csharp
public struct Address
{
    public int Process;  // 进程 ID
    public int Fiber;    // 纤程 ID
}

public struct ActorId
{
    public Address Address;
    public long InstanceId;  // Entity.InstanceId，对应 MailBoxComponent 的 Parent
}
```

- **Address**：唯一标识一个 Fiber。
- **ActorId**：唯一标识一个可收消息的 Entity（即 Actor）。
- 发往 `InstanceId = 1` 的消息通常发往 Fiber 的 Root Scene。

### 3.2 ProcessInnerSender

进程内 Fiber 间通信的唯一通道：

```csharp
// 发送单向消息
ProcessInnerSender.Send(actorId, message);

// 发送请求并等待响应（RPC）
IResponse response = await ProcessInnerSender.Call(actorId, request);
```

`Call` 行为：

1. 分配 RpcId，设置 request.RpcId。
2. 将 request 投递到目标 Fiber 的 `MessageQueue`。
3. 注册回调，等待 Reply。
4. 超时（默认 40 秒）未收到响应则触发超时逻辑。

### 3.3 MessageQueue

```csharp
// 每个 Fiber 对应一个 ConcurrentQueue<MessageInfo>
public bool Send(Address fromAddress, ActorId actorId, MessageObject message)
{
    if (!this.messages.TryGetValue(actorId.Address.Fiber, out var queue))
        return false;  // 目标 Fiber 不存在
    queue.Enqueue(new MessageInfo { ActorId = ..., MessageObject = message });
    return true;
}
```

- 按 **FiberId** 分配队列，消息进入目标 Fiber 的队列。
- `ProcessInnerSender.Update` 中从队列取消息，再根据 `ActorId.InstanceId` 路由到对应 Actor。

### 3.4 消息处理流程

```
Send/Call
    → MessageQueue.Send(fiberId, message)
    → ProcessInnerSender.Update() 取出消息
    → Mailboxes.Get(actorId.InstanceId) 得到 MailBoxComponent
    → mailBoxComponent.Add(fromAddress, message)
    → 按 MailBoxType 分发：EventSystem.Invoke(MailBoxType, MailBoxInvoker)
    → MailBoxType_XXXHandler 调用 MessageDispatcher.Handle
    → 找到 MessageHandler，执行 Run(entity, message)
    → 若为 IRequest，执行完毕 Reply 回发者
```

### 3.5 MailBox 类型

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **UnOrderedMessage** | 无顺序约束，直接分发给 Handler | Scene 级 Actor（Gate、Realm、Map 等） |
| **OrderedMessage** | 带 CoroutineLock，保证同一 Actor 消息顺序处理 | Unit、Player 等需严格顺序的 Entity |
| **GateSession** | 收到消息后转发给客户端 Session | Gate 上与客户端绑定的 Session |

## 四、Fiber 初始化示例

### 4.1 服务端 Gate Fiber

```csharp
[Invoke((long)SceneType.Gate)]
public class FiberInit_Gate : AInvokeHandler<FiberInit, ETTask>
{
    public override async ETTask Handle(FiberInit fiberInit)
    {
        Scene root = fiberInit.Fiber.Root;
        root.AddComponent<MailBoxComponent, MailBoxType>(MailBoxType.UnOrderedMessage);
        root.AddComponent<TimerComponent>();
        root.AddComponent<CoroutineLockComponent>();
        root.AddComponent<ProcessInnerSender>();
        root.AddComponent<MessageSender>();
        root.AddComponent<PlayerComponent>();
        // ...
    }
}
```

### 4.2 客户端 NetClient Fiber

```csharp
[Invoke((long)SceneType.NetClient)]
public class FiberInit_NetClient : AInvokeHandler<FiberInit, ETTask>
{
    public override async ETTask Handle(FiberInit fiberInit)
    {
        Scene root = fiberInit.Fiber.Root;
        root.AddComponent<MailBoxComponent, MailBoxType>(MailBoxType.UnOrderedMessage);
        root.AddComponent<TimerComponent>();
        root.AddComponent<CoroutineLockComponent>();
        root.AddComponent<ProcessInnerSender>();
        root.AddComponent<FiberParentComponent>();
    }
}
```

- Root Scene 默认有 `InstanceId = 1`，挂在 Root 的 `MailBoxComponent` 使 Root 成为 Actor。
- 发往 `new ActorId(process, fiberId)` 的消息会到达该 Fiber 的 Root。

## 五、MessageHandler 与消息分发

### 5.1 MessageHandler 定义

```csharp
[MessageHandler(SceneType.NetClient)]
public class Main2NetClient_LoginHandler : MessageHandler<Scene, Main2NetClient_Login, NetClient2Main_Login>
{
    protected override async ETTask Run(Scene root, Main2NetClient_Login request, NetClient2Main_Login response)
    {
        // 在 NetClient Fiber 中执行
        // ...
        response.PlayerId = g2CLoginGate.PlayerId;
    }
}
```

- `[MessageHandler(SceneType.XXX)]`：该 Handler 处理发往 `SceneType.XXX` 类型 Fiber 的消息。
- `MessageDispatcher` 根据消息类型查找 Handler，并调用 `Handle(entity, fromAddress, message)`。

### 5.2 请求-响应模式

- 继承 `MessageHandler<E, Request, Response>`：Handler 执行完后自动 `Reply(fromAddress, response)`。
- 继承 `MessageHandler<E, Message>`：单向消息，无 Reply。

## 六、登录流程中的 Fiber/Actor 应用

以 `Main2NetClient_LoginHandler` 为例说明跨 Fiber 调用：

```
Main Fiber (ClientSenderComponent.LoginAsync)
    → FiberManager.Create(SceneType.NetClient)  创建 NetClient Fiber
    → ProcessInnerSender.Call(netClientActorId, Main2NetClient_Login)
        → 消息进入 NetClient 的 MessageQueue
        → NetClient Fiber 的 ProcessInnerSender.Update 取出
        → Mailboxes.Get(1) 得到 Root 的 MailBox
        → MessageDispatcher 找到 Main2NetClient_LoginHandler
        → 在 NetClient Fiber 中执行 Run()
    → Reply(NetClient2Main_Login) 回到 Main Fiber
    → ClientSenderComponent.LoginAsync 的 await 返回
```

**为何必须用消息而不是直接调用？**

1. **Fiber 隔离**：NetClient 的 `NetComponent`、`Session` 等挂在 NetClient Scene，Main 无法直接访问。
2. **线程安全**：NetClient 可能跑在 ThreadPool 的不同线程，直接调用会破坏单线程假设。
3. **架构统一**：进程内、进程间都走消息，便于理解和扩展。

## 七、Actor 死锁与规避

若出现 **A Call B → B Call C → C Call A** 的环路，且都在同一 MailBox 顺序处理中，则会死锁。

规避方式：

- 在 Handler 内 **新开协程** 处理耗时逻辑，不阻塞 MailBox 队列：

```csharp
protected override ETTask Run(Unit unit, Actor_Test message)
{
    RunAsync(unit, message).Coroutine();  // 不 await，立即返回
    return ETTask.CompletedTask;
}
```

- 避免形成环状 Call 链，或改用 Send 替代部分 Call。

## 八、总结

| 概念 | 说明 |
|------|------|
| **Fiber** | 最小并发单元，拥有独立 Scene、Mailboxes、ThreadSynchronizationContext |
| **Actor** | 挂有 MailBoxComponent 的 Entity，可按 InstanceId 接收消息 |
| **ProcessInnerSender** | 进程内 Fiber 间消息与 RPC 的唯一通道 |
| **MessageQueue** | 按 Fiber 维度的消息队列 |
| **MailBoxType** | 决定消息处理顺序与是否转发（如 GateSession） |
| **调度器** | Main / Thread / ThreadPool，控制 Fiber 在哪些线程执行 |

ET 的 Fiber + Actor 模型实现了：

1. **逻辑单线程**：每个 Fiber 内顺序执行，无锁开发。
2. **安全并发**：跨 Fiber 仅通过消息，避免共享状态竞争。
3. **统一抽象**：进程内、进程间、客户端网络层使用同一套消息机制。
